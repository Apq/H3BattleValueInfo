# 战斗面板背景图渲染：DirectDraw Blt 方案

## 问题背景

HD_TC2（HD mod 真彩模块）hook 了游戏原始的 `_Pcx8_::DrawToPcx16`（`0x44FA80`）。
自定义创建的 `_Pcx8_` 对象（即使注册到游戏资源树）在 HD_TC2 hook 内部会触发
`EXCEPTION_ACCESS_VIOLATION`（read of address `0x3BC`），导致游戏崩溃。

直接写 `screen_pcx16->buffer` 也不可行：HD_TC2 改变了屏幕渲染管线，
`DrawTextToPcx16`（`0x4B51F0`）等游戏函数被 hook 后渲染到 HD_TC2 自己的 32-bit
buffer，手动写入 screen_pcx16 的 16-bit 像素不会正确显示（颜色和位置都会错乱）。

## 根因分析

1. `_Pcx8_::DrawToPcx16`（`0x44FA80`）被 HD_TC2 hook
2. HD_TC2 的 hook 内部访问 `_Pcx8_` 对象偏移 `0x3BC` 处的字段
3. 自定义 `_Pcx8_`（通过 `_Pcx8_::CreateNew` 创建）缺少 HD_TC2 期望的内部状态
4. 即使通过 `o_BinTreeHead.AddItem()` 注册到资源树也无法解决

## 解决方案：DirectDraw 离屏 Surface + Blt

完全绕开 `_Pcx8_`/`_Pcx16_` 体系，直接在 DirectDraw 层面操作：

```
PCX 文件 → libpng/手动解码 → RGB 像素
  → o_DD->CreateSurface() 创建离屏 DirectDraw surface
  → Lock surface，按实际像素格式（16-bit RGB565 或 32-bit BGRA）写入
  → o_DDSurfaceBackBuffer->Blt() 画到后备缓冲区
```

### 关键代码路径

1. **解码 PCX**：读取 1-plane 8-bit PCX，提取索引像素 + 768 字节调色板
2. **创建 DD Surface**：
   ```cpp
   DDSURFACEDESC ddsd;
   ddsd.dwFlags = DDSD_CAPS | DDSD_WIDTH | DDSD_HEIGHT;
   ddsd.ddsCaps.dwCaps = DDSCAPS_OFFSCREENPLAIN;
   o_DD->CreateSurface(&ddsd, &dd_surf, nullptr);
   ```
3. **Lock + 写像素**：检测 `DDPIXELFORMAT.dwRGBBitCount`（16 或 32），
   分别用 RGB565 或 BGRA32 写入
4. **Blt 到 backbuffer**：
   ```cpp
   o_DDSurfaceBackBuffer->Blt(&dst_rect, dd_surf, &src_rect, DDBLT_WAIT, nullptr);
   ```

### 需要的头文件

- `#include <ddraw.h>` 必须在 `#include "homm3.h"` 之前
- `homm3.h` 中 `ddraw.h` 默认被注释掉了

## 不工作的方案（记录避免重复踩坑）

| 方案 | 结果 | 原因 |
|------|------|------|
| `_Pcx8_::DrawToPcx16`（自定义 Pcx8） | 崩溃 | HD_TC2 hook 内访问无效偏移 |
| 手动 BlitPcx8ToPcx16（直接写 screen buffer） | 颜色/位置错乱 | HD_TC2 渲染管线覆盖了 screen_pcx16 |
| 注册到 `o_BinTreeHead` 资源树 | 崩溃 | HD_TC2 检查的不只是资源树归属 |
| 平铺原版 DlgBluBk.PCX | 可用但非目标背景 | 只能用原版蓝色纹理 |
| `_Pcx16_::DrawPcx16`（我们的 C++ 实现） | 同手动写 buffer | 也是直接写 screen_pcx16->buffer |

## 相关全局变量

- `o_DD`（`0x6AAD20`）：LPDIRECTDRAW
- `o_DDSurfacePrimary`（`0x6AAD24`）：主表面
- `o_DDSurfaceBackBuffer`（`0x6AAD28`）：后备缓冲区
- `o_WndMgr->screen_pcx16`：游戏逻辑层面的屏幕表面（HD_TC2 下不可直接写）
